# 핫딜 연구소 (HOTDEAL LAB)

**AI 기반 실시간 커뮤니티 핫딜 큐레이션 및 시맨틱 추천 플랫폼**
서버리스(Serverless) 아키텍처 위에서 데이터 수집, AI 분석, Vector DB 검색이 매끄럽게 연결되는 풀스택 프로젝트임.

---

## 1. 기획 배경 및 문제 정의

**"쏟아지는 정보 속에서, 나에게 진짜 필요한 핫딜만 골라볼 수는 없을까?"**
기존 커뮤니티의 핫딜 게시판은 하루에도 수백 개의 글이 올라오며, 이 중 상당수는 나와 무관한 제품이거나 교묘한 바이럴/광고성 글임. 사용자는 원하는 할인 정보를 찾기 위해 불필요한 피로도를 느껴야 했음.

이에 **단순한 키워드 알림을 넘어, AI가 문맥을 이해하고 가치를 판단하여 나만의 맞춤형 핫딜을 추천해 주는 서비스**를 기획하게 됨.

---

## 2. 핵심 도입 기술 및 AI 활용 기댓값

본 프로젝트는 기획 단계부터 데이터 정제와 검색 경험 향상을 위해 **Google Gemini AI 생태계**를 전략적으로 도입함.

### 2-1. 데이터 파이프라인 정제 (`gemini-3-flash`)
* **도입 배경:** 수많은 미가공 핫딜 데이터 중 광고성 글 혹은 스팸성 글을 구분하는 것은 단순한 크롤링만으로는 한계가 있었음.
* **원리 및 효과:** 백그라운드 크롤러가 수집한 데이터를 AI가 즉각 분석함. 유저들의 댓글 뉘앙스를 파악해 바이럴 스팸을 차단하고, 가치 있는 상품에 매칭 점수와 1줄 요약(`ai_summary`)을 부여하여 **데이터베이스의 신뢰도를 극대화**함. 이 과정에서 네이버 최저가 정보를 가져와 비교하는 로직도 포함됨.

### 2-2. AI 기반 시맨틱(Semantic) 검색 및 맞춤형 추천
* **도입 배경:** 기존의 단순 키워드 검색(`ilike`)은 사용자가 정확한 단어를 입력하지 않으면 결과를 찾지 못하는 한계(Low Recall)가 있었음. 예를 들어 "주말 캠핑용 고기 추천해줘" 같은 명확하지 않은 일상어 요청에 대해서도 시스템이 문맥을 이해하고 매칭할 수 있어야 했음.
* **원리 및 효과 (데이터 저장 ~ 검색):** 
  1. **데이터 저장 (`gemini-embedding-001`):** 상품명, 카테고리 등의 텍스트를 AI가 768차원의 수학적 벡터(Vector)로 변환해 Supabase Vector DB에 미리 저장함. (의미론적 데이터 공간 구축)
  2. **조회 및 추천 (`gemini-2.5-flash`):** AI가 유저의 문맥을 분석해 검색 의도를 벡터로 변환한 뒤, DB에 적재된 핫딜 벡터들과 **코사인 유사도(Cosine Similarity)**를 연산함. 거리가 가장 가깝고 문맥이 일치하는 최상위 결과를 도출함으로써, 사용자는 딱딱한 쿼리 없이 사람과 대화하듯 원하는 핫딜을 찾을 수 있음.

### 2-3. 무중단 서비스 폴백 아키텍처 (`gemini-2.5-flash-lite`)
* **도입 배경:** AI API 특성상 트래픽이 몰릴 경우 Rate Limit(429 Error)이 발생하여 서비스 경험이 끊길 위험이 존재했음.
* **원리 및 효과:** 메인 추론 모델 응답 지연/실패 감지 시, 에러를 뱉는 대신 **즉각적으로 빠르고 가벼운 백업 모델(Flash-Lite)로 트래픽을 우회(Fallback)**하는 로직을 설계함. 이를 통해 서버리스 환경에서도 **100%에 가까운 가용성과 응답성을 보장**함.

## 3. 트러블슈팅 

### 3-1. 키워드 검색의 한계 돌파 (RDBMS `ilike` → Vector DB 마이그레이션)
* **Problem:** 초기 개발 시에는 AI가 유저 검색어에서 대표 키워드 배열을 뽑아내어 RDBMS의 `ilike OR` 쿼리로 매칭했음. 하지만 '소시지'를 검색하면 '핫도그' 핫딜을 놓치고, 너무 포괄적인 단어는 관련 없는 데이터를 가져오는 치명적 한계가 있었음.
* **Action:** 텍스트를 고차원 수치로 변환하는 `gemini-embedding-001`을 도입하고, Supabase에 `pgvector` 확장을 적용해 **코사인 유사도 검색 전용 RPC 함수**를 구축함.
* **Result:** 동의어나 은유적 표현("다이어트 간식")에도 찰떡같이 반응하는 높은 품질의 검색 시스템으로 진화했으며, RDBMS와 Vector Search의 패러다임 차이를 완벽히 체감함.

### 3-2. AI API Rate Limit(RPD/RPM) 최적화 및 모델 분배 전략
* **Problem:** 초기 서비스 환경에서는 AI 모델의 분당 요청 제한(RPM)과 일일 요청 제한(RPD)으로 인한 병목 현상이 문제였음. 실시간 크롤링 정제, 벡터 변환, 유저 검색이 동시에 발생하면 429(Too Many Requests) 에러로 인해 시스템이 마비될 위험이 컸음.
* **Action:** 타겟 기능에 맞춰 최적의 모델을 분배하고, API 호출 횟수를 최소화하기 위해 다중 요약을 묶는(Batch) 로직을 도입함.
  1. **크롤링 정제 (`gemini-3-flash`):** 백그라운드에서 수집되는 수많은 핫딜을 건당 처리하면 RPM이 초과됨. 이를 방지하기 위해 핫딜 데이터를 **최대 10~20개 단위로 묶어서(Batch)** 하나의 프롬프트로 전송하여 한 번에 분석 결과를 배열 형태로 반환받음. 결론적으로 API 호출 횟수를 획기적으로 낮춤.
  2. **벡터 변환 (`gemini-embedding-001`):** 분석이 끝난 텍스트 데이터 역시 개별 호출이 아닌 배열(Array) 단위로 묶어 한 번에 768차원 임베딩 값으로 치환 후 DB에 적재함.
  3. **유저 추천 (`gemini-2.5-flash`):** 실시간 문맥 파악이 중요한 검색 영역에는 지연 시간이 짧고 품질이 높은 모델을 단건으로 사용해 유저 체감 속도를 높임.
  4. **우회 폴백 (`gemini-2.5-flash-lite`):** 사용자 트래픽 급증 시 메인 모델이 Rate Limit에 도달하면 즉각 가장 가볍고 할당량이 넉넉한 Lite 모델로 요청을 돌리도록 예외 처리(Try-Catch Fallback)를 구축.
* **Result:** 제공되는 한정된 API 가용 자원(RPD/RPM) 내에서도 막힘 없는 수집 파이프라인과 안정적인 서비스 응답을 보장하는 고효율 아키텍처를 완성함.

## 4. 시스템 아키텍처 개요 

* **프론트엔드 (Frontend Layer):** Next.js 15 (App Router) 기반 초고속 렌더링. Framer-Motion을 이용한 부드러운 UI 전환 및 React Swiper 연동. SPA(Single Page Application) 경험 극대화를 위한 컴포넌트 이벤트 기반 상태 관리.
* **백엔드 (Backend & BaaS):** Next.js Server Actions를 활용한 안전하고 빠른 API 통신 구조 설계 및 Supabase (PostgreSQL + pgvector) 연동.
* **크롤링 (Data Pipeline):** Python 기반 독립 동작 크롤러. 정규화 및 AI 1차 분류 후 DB 적재. 
> **크롤러 시스템 상세 :** [HOTDEAL_SCRAPER Repository](https://github.com/RohUkJin/HOTDEAL_SCRAPER)

## 5. 기술 스택 
* **Client:** Next.js 15, React(TypeScript), Styled-Components, Framer Motion
* **Backend:** Next.js Server Actions 
* **DB/BaaS:** Supabase (PostgreSQL + pgvector)
* **AI:** Google Gen AI SDK (Gemini)
